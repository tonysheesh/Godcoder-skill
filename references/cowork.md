# CoWork Mode Protocol

CoWork mode turns Godcoder into a self-training agent that drives GUI/OS
applications — starting with Open Cowork, but adaptable to any desktop target.

The key distinction from Harness mode: CoWork **executes human-action tasks**.
For any step a person would do at a keyboard or screen, CoWork gets an actuation
plan and carries it out via GUI automation, verifying each step with a screenshot.

---

## Actuation Classifier

Before each planned action, classify it into one of three segments:

| Segment | Definition | Agent handles? |
|---|---|---|
| **Digital** | Pure computation, file I/O, API calls, data transformation | ✅ Yes — use standard tools |
| **Actuatable** | GUI interaction a human would do at keyboard/screen | ✅ Yes — use `cowork_act` |
| **Physical** | Requires a body: drive, lift, repair, in-person signature | ❌ No — hand back to user |

Only physical-world steps are handed back. Everything else the agent handles.

---

## CoWork Loop

Same structure as the harness loop, with two additions in EXECUTE:

```
EXECUTE
  ├── Digital steps → standard file/bash/API tools
  ├── Actuatable steps → cowork_act(plan) + cowork_screenshot() verify
  └── Physical steps → pause, ask user, resume when confirmed
```

Verification for actuatable steps always uses `cowork_screenshot()` to confirm
the UI reached the expected state before proceeding.

---

## Tool Extensions (CoWork only)

### cowork_act(plan)
Execute a GUI/OS actuation plan. The plan is a structured sequence of actions:

```json
{
  "steps": [
    {"action": "click", "target": "selector or description", "verify": "element state after"},
    {"action": "type", "text": "content to type", "target": "input field description"},
    {"action": "key", "keys": ["Ctrl", "S"]},
    {"action": "wait", "condition": "UI state to wait for", "timeout_ms": 3000},
    {"action": "open_app", "name": "application name"},
    {"action": "send_email", "to": "...", "subject": "...", "body": "..."}
  ]
}
```

Returns: execution result for each step (success/failure + observed state).

### cowork_screenshot()
Capture the current screen state.
Returns: screenshot + optional OCR text extraction.
Use after every actuatable step to verify the UI reached expected state.

## Sandbox

CoWork sandbox: `cowork-build/`

Contains:
- `cowork-build/COWORK_LOG.md` — human-readable iteration log
- `cowork-build/objectives/` — objective files and progress tracking
- `cowork-build/training-data/` — captured interaction patterns
- `cowork-build/scripts/` — reusable automation scripts generated by the agent

---

## Objective Handling

On start, if user provided a prompt:
1. Parse it into a structured objective with sub-tasks.
2. Classify each sub-task (digital / actuatable / physical).
3. Emit the classified plan for user review before executing.
4. Execute after approval (or immediately if user said "just go").

If no prompt (pure self-training):
1. Open Open Cowork.
2. Route to the highest-value capability to learn next (via `memory_route`).
3. Attempt to use that capability end-to-end.
4. Log outcome, optimize, repeat.

---

## Memory Tags for CoWork

Always use `cowork:` prefix on all CoWork memory entries. Additional tags:

| Tag | When to use |
|---|---|
| `cowork:pptx` | PPTX skill interactions |
| `cowork:docx` | DOCX skill interactions |
| `cowork:xlsx` | XLSX skill interactions |
| `cowork:pdf` | PDF skill interactions |
| `cowork:mcp` | MCP connector interactions |
| `cowork:actuation` | GUI automation steps |
| `cowork:email` | Email-related actuation |
| `cowork:esign` | E-signature actuation |

---

## Failure Handling

If a `cowork_act` step fails:
1. Capture screenshot of failure state.
2. Log failure with screenshot description as evidence.
3. Attempt recovery: re-route around the failed step if possible.
4. If unrecoverable: checkpoint restore → log → re-route to different action.
5. Never retry the exact same action sequence more than 2 times without modifying it.
